01How it was instruction-tuned

The base model is a completer — it continues text but cannot answer a question. To make it follow instructions, we built a Q&A dataset from the very same corpus it was pretrained on, using GPT-4.1-mini as a teacher, then filtered it hard.

  1. Chunk the corpusSplit cleaned documents into 2,500 passages of ~500 tokens (measured with the base model's own tokenizer, so every example provably fits the 1,024-token window alongside its question and answer).
  2. Synthesize with a teacherGPT-4.1-mini writes 3 pairs per passage across four task types, at varied temperature for diversity. Every answer must be grounded in its passage; nothing may be invented.
  3. LLM-as-judgeA second, stricter pass scores each pair 1–5 for correctness and grounding. Anything below 4 is discarded (139 dropped).
  4. Filter hardToken-overlap grounding check → near-duplicate removal → decontamination: any pair sharing a 13-word span with the CaseHOLD/LexGLUE benchmarks is deleted, so evaluation stays honest.
  5. Train3 epochs on a single H100, lr 2e-5 cosine, AdamW, bf16. Crucially, loss is computed on the assistant's answer only — the context and question are masked out, so the model learns to answer, not to parrot prompts.

02The pretraining data

Three public datasets streamed from HuggingFace and filtered to a 2.04-billion-token corpus. The two legal sources are small — together they cap at ~2B tokens — so the strategy is legal-first: take essentially all of the legal data, add a thin web slice for fluency. This is not a 70/20/10 split.

03The BPE tokenizer

Models read tokens, not characters. This one uses a byte-level Byte-Pair Encoding tokenizer with a 16,384-token vocabulary, trained from scratch on the legal corpus so its merges reflect legal language.

  • Start from raw bytes. The base alphabet is the 256 possible bytes, so any UTF-8 text is representable and there is effectively no “unknown token.”
  • Merge greedily. Training repeatedly merges the most frequent adjacent pair, growing the vocab from 256 to 16,384. Frequent legal phrases collapse into single tokens.
  • Why only 16K? The embedding table is 16,384 × 768 ≈ 12.6M params — already ~10% of the model. A bigger vocab would spend the tiny parameter budget on embeddings instead of transformer layers. It still averages ~3.9 characters per token here.
  • Reserved tokens. <|bos|> <|eos|> <|pad|> <|unk|> plus <|system|> <|user|> <|assistant|> — the chat markers the fine-tune later builds its template on.

04The architecture

A decoder-only Transformer in the Llama style: pre-norm RMSNorm, rotary embeddings (RoPE), SwiGLU feed-forward blocks, and input/output embeddings tied. Twelve blocks stacked.

06Variations that could have been tried

  • More refusal data. The clearest weakness. Refusals were 17% of the fine-tuning set; pushing that to ~30% (plus harder negatives) is the most direct fix for the hallucination shown above.
  • More pretraining epochs. Perplexity was still falling when pretraining stopped at 1 epoch. 3–5 epochs would give the fine-tune a stronger base to start from.
  • Grouped-Query Attention. Fewer KV heads than query heads shrinks the KV cache and speeds inference, at a small quality cost.
  • Bigger model. 350M–1B would hold far more knowledge and handle the "is this answerable?" judgment far better. Scale is the main lever for both.
  • Larger vocab or longer context. 32K vocab shortens sequences; a 2,048-token window fits longer legal passages.
  • RAG. The model knows no facts of its own. Retrieval would ground it in real documents — the single biggest upgrade beyond fine-tuning.

What this is instruction-tuned

Built end-to-end from nothing: streamed and cleaned a legal corpus, trained a tokenizer, pretrained a 125M model from random weights on 8×H100, then synthesized and filtered a Q&A set and fine-tuned it to follow instructions. Total compute cost: ~$15.

It has no knowledge of its own — at 125M parameters a model holds only a few tens of MB of facts. It works over the context you give it, and it will sometimes get that wrong. Not legal or financial advice.
